Architecture Surface

Context boundaries are the retrieval system.

ExoCortex does not dump everything into one global memory bucket. The folder you start in determines the boundary, the visible contract packet, and the truth you should inherit first.
Core Rule

Project truth stays project-local. Domain patterns stay domain-local. Root promotion happens only when it is genuinely cross-context.

Repository Surface
Repo root Broad routing, onboarding, cross-domain coordination.
system/ Control plane for persona, rules, priorities, and health.
agents/ Stable execution roles rather than endless new personas.
domains/ → projects/ The main truth boundaries for execution.
journal/ • wiki/ • raw/ Durable outputs, managed synthesis, and near-original source material.

Visible Contract Packet
  • README.md tells the system what this context is for.
  • AGENT.md shapes how the active role should behave here.
  • STATE.md carries what is currently true right now.
  • MEMORY.md and WORKFLOWS.md carry durable local learning.
  • Ancestor context is loaded as a smaller supporting packet, not as a replacement for the local truth.
What The Boundary Controls
1. Which files come into view first

The wrapper loads local contracts plus the smallest relevant ancestor surface, so retrieval starts coherent instead of global and noisy.

2. Which agent and mode become the default

Stable roles and operating modes are resolved from the active context rather than guessed ad hoc every session.

3. Where future signal should land

Useful output does not just “exist in chat.” It lands in journal/, wiki/, or promoted local contracts at the right level.

4. What should stay local vs become cross-context

This is how ExoCortex keeps inspectable memory without turning the whole repo into an undifferentiated retrieval dump.
